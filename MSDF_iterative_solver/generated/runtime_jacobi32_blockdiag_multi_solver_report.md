# Runtime Jacobi32 Block-Diagonal Multi-Iteration Solver Report

This report records the first multi-iteration solver checkpoint for the
runtime-loadable iterative-solver top.

## Scope

| item | value |
| --- | --- |
| Matrix shape | `32 x 32` |
| Physical mapping | `8` clusters x `4` rows/cluster |
| Active rows | `32` |
| Degree per row | `4` fixed slots |
| Iterations checked | `6` |
| Replay mode | committed ping-pong state replay |
| Replay digit | `DATA_WIDTH - 1`, i.e. LSB under current MSB-first replay selector |
| Runtime storage | `runtime_mem_style=1`, RAM-wrapper field banks |
| Initial state | all-zero runtime-loaded state |

The fixture is generated by:

```bash
conda run -n qas python MSDF_iterative_solver/make_jacobi32_blockdiag_multi_runtime_vectors.py
```

Generated vectors are stored under:

```text
MSDF_iterative_solver/generated/rtl_vectors/jacobi32_blockdiag_multi/
```

## What This Adds Over the Single-Iteration Checkpoint

The previous `jacobi32_blockdiag` test checked one runtime-loaded iteration.
This checkpoint verifies the solver-level loop:

```text
start iteration
-> replay committed state digit
-> issue all 32 rows
-> row update / delta / block_H certification
-> wait for all clusters
-> commit ping-pong state
-> use committed state as next iteration input
```

The golden model follows the current RTL observable contract. It does not
compare against a full floating-point Jacobi solve. This is intentional because
the current RTL replays one selected rail digit per iteration.

## Current Limitation

This is now a real multi-iteration runtime solver checkpoint, but it is still
cluster-local and digit-slice based. The next architectural gap is unchanged:
generic banded `32 x 32` Jacobi requires inter-cluster source routing, preferably
via a halo-window mechanism before considering a full cluster crossbar.
