This report records the bounded inter-cluster source-routing checkpoint for the
runtime-loadable iterative-solver top.


## Why Halo-Window

The raw banded `32 x 32` fixture has maximum cross-cluster dependency radius of
one cluster. A full global-source path lets every cluster select from all `32`
active rows, which is functionally general but physically expensive. Halo-window
replay instead restricts each cluster to a fixed source window:

```text
[previous cluster rows][current cluster rows][next cluster rows]
```

For `4` rows/cluster and halo radius `1`, this reduces replay source rows from
`32` to `12` and source index width from `5` to `4` bits while preserving the
same raw banded Jacobi semantics for this workload.

Relative to global-source replay, the final registered halo-window path reduces
LUT by `43.54%`, dynamic power by `51.87%`, total power by `19.33%`, and
improves WNS by `1.097 ns`. Relative to the unregistered halo-window path, it
improves WNS by `0.857 ns`, reduces LUT by `0.88%`, reduces dynamic power by
`10.02%`, and increases FF by `2.11%`. It does not change solver-level cycle
count because the existing runtime scheduler already holds replay control
stable before issuing row updates.

The NC16 scaling checkpoint routes `64` active rows at 5 ns. The product
pipeline is not useful: adding `cert_product_pipeline=1` improves WNS by only
`0.019 ns` while increasing LUT by `3.22%`, FF by `8.78%` and CARRY8 by
`25.67%`. The operand-localization pipeline is the useful scaling option:
`cert_operand_pipeline=1` improves NC16 WNS from `+0.390 ns` to `+0.646 ns`,
keeps LUT/FF/CARRY8 essentially unchanged, and lowers dynamic power from
`1.766 W` to `1.713 W`. The cost is one extra certification wait cycle per
iteration, so it should be enabled for larger active-cluster scaling or tighter
clock closure, not for the smallest latency checkpoint.

The compare pipeline is not useful after operand localization. Adding
`cert_compare_pipeline=1` on top of opipe keeps NC16 WNS at `+0.646 ns`, but
increases FF by `750`, raises dynamic power by `0.015 W`, and adds another
certification wait cycle. It is kept only to document the negative result.

## Interpretation

Halo-window replay is the first practical inter-cluster routing candidate. It
keeps the solver-level semantics of the raw banded `32 x 32` Jacobi fixture, but
avoids the full all-to-all source mux of the global-source checkpoint.

The final registered halo-window path with a kept block-bound pipeline stage
and slot-based template cache should replace global-source replay as the default
path for banded workloads. The extra synthesis attributes are not just cosmetic:
without keeping that boundary, Vivado can optimize through the intended
`block_bound_max_pool -> cert_engine` register cut and the worst path enters the
certification DSP input. The slot-based template cache then reduces the window
cache fanout enough to reach `+1.125 ns` WNS at 5 ns. The remaining worst path
is still template-cache control route, not replay or certification arithmetic.

At NC16, the bottleneck shifts from template-cache control to `block_H`
certification. The product pipeline removes the product-to-sum portion, but the
routed worst path remains from `r_block_bounds_reg` into a DSP multiplier output
register, dominated by block-bound-to-DSP routing and DSP internal multiply
delay. The operand-localization pipeline addresses the actual issue by first
replicating and registering `{block_bound, block_weight}` per product term
inside `cert_engine`, then letting those local operands drive the DSPs. After
that change, a further compare split does not improve timing. The dominant path
moves back to template-window cache routing, so the next real optimization
target is the runtime template window cache/control fanout rather than additional
certification arithmetic pipelining.
